## 2. Persistence Readiness 设计轮

要解决的问题：

- Data Model、Workflow State 和 Execution Contract 已经定义了数据、状态和模块契约，但还缺少“怎么安全落到 SQLite”。
- 需要明确 Repository / DAO 边界、Unit of Work、事务边界、Project open gate、migration readiness、recovery query、idempotency query。
- 需要确保 FakeProvider 单页后端切片能使用真实临时 SQLite 文件和真实 workspace 文件，而不是内存假对象。

实际做了什么：

- 建立 persistence 设计包：
  - `docs/design/persistence/GOAL.md`
  - `docs/design/persistence/HARNESS.md`
  - `docs/design/persistence/PLAN.md`
  - `docs/design/persistence/open-questions.md`
- 生成并评审 persistence proposals，覆盖 repository boundary、Unit of Work、migration/db lifecycle、recovery/idempotency、FakeProvider readiness。
- 合成最终设计：
  - `docs/design/persistence/final/persistence-readiness-dd-v0.1.md`
  - `docs/design/persistence/final/repository-contract-minimal.md`
  - `docs/design/persistence/final/unit-of-work-and-transactions.md`
  - `docs/design/persistence/final/migration-strategy-minimal.md`
  - `docs/design/persistence/final/fakeprovider-persistence-readiness.md`
  - `docs/design/persistence/final/open-questions.md`

关键结论：

- `app.db` 只放全局 Project registry、非 secret app settings、provider/profile 骨架和 app migration ledger。
- 每个 Project 一个 `project.db`，放 Batch、Page、TextBlock、OCR/Translation result、workflow evidence、artifact metadata、quality issue、profile snapshot。
- Project-scoped repositories 只能在 Project identity 和 migration readiness 通过后暴露。
- Repository / DAO 是唯一 SQLite 访问入口，上层不能拿 SQL、cursor、ORM session 或 table-shaped row dict。
- StageExecutor 只能使用窄的 `StageEvidenceWriter` 写 ToolRunLog 和 attempt evidence，不能写 active pointer、QualityIssue、WorkflowDecision 或 retry budget。
- QualityCheckService 在 MVP-0 中保持 repository-free，只返回 issue draft / report。
- Provider call 期间不能持有 SQLite write transaction。
- Acceptance transaction 是语义提交点，只有它能让 OCRResult、TranslationResult、cleaned/typeset artifact pointer 变成当前有效结果。
- `Page.status` 只是摘要，recovery 不能只靠它；恢复必须读取 task、attempt、decision、tool log、active pointer、artifact state、quality issue、dependency hash 等证据。
- MVP-0 persistence 停在 `ready_for_export`；实际 ExportRecord、ZIP、manifest 和 export output 后置。

结果：

- 项目已经具备开始 MVP-0 Slice 01 的 persistence 设计输入。
- 后续实现不需要先发明完整 ORM 或迁移框架，只需要按 slice 逐步实现最小可验证边界。

## 6. Prompt-pattern 轻量结构

要解决的问题：

- Data Model 设计轮使用了较完整的多 agent prompt-pattern，这个经验有复用价值。
- 但不能把 Data Model 的重流程硬套到所有设计任务，也不能为了 prompt-pattern 造 custom harness runtime、agent loop、script runner、hooks system 或 skill framework。
- 需要保留成功案例，同时抽出轻量可复用模板。

实际做了什么：

- 建立 prompt-pattern 目录说明：
  - `docs/prompt-patterns/README.md`
- 保留 Data Model 详细设计协作作为 case study：
  - `docs/prompt-patterns/case-studies/data-model-detailed-design.md`
  - `docs/prompt-patterns/case-studies/data-model-detailed-design-template.md`
- 抽出详细设计通用模式：
  - `docs/prompt-patterns/design/detailed-design-pattern.md`
- 抽出 implementation slice task 通用模式：
  - `docs/prompt-patterns/implementation/slice-task-pattern.md`

保留的 Data Model 经验：

- `GOAL -> HARNESS -> proposal agents -> cross-review -> limited revision loop -> synthesizer -> harness validation -> ADR / open questions`。
- role-biased proposal agents。
- P0/P1/P2 分类。
- hard invariants。
- 用 HARNESS scenarios 验证设计。
- cross-review 专注找冲突、遗漏、over-design、under-design 和 blocking issue。
- limited revision loop 防止无限争论。
- synthesizer 做裁决，不是拼文档。
- validation agent 和 synthesizer 分离。

泛化出的 detailed-design pattern：

- 适用于 API design、UI flow design、export design、real tool integration design、testing strategy design 等后续详细设计。
- 明确 detailed-design prompt-pattern 只是写真实 prompt 的模板，不是真实 prompt 本身。
- HARNESS 不是更长的 prompt，而是定义设计必须证明什么。
- 如果 scenario 不能被设计证据证明，要记录为 open question 或 implementation validation need。

泛化出的 implementation slice pattern：

- 基于 MVP-0 Slice 01-07 的结构。
- 规定每个 implementation prompt 应包含 Goal、Source documents、Allowed files、Forbidden files、Implementation boundaries、Validation command、Expected output、Commit rule、Stop conditions、Final report requirements。
- 明确真正的 implementation harness 是 tests、commands、diffs、file boundaries 和 reviewable evidence。
- 明确优先用 pytest 和 focused integration tests，不造新 harness runtime。
- 明确 pattern 不能把 slice 扩大到 slice 文档之外。

结果：

- 项目拥有了轻量 prompt-pattern 资产。
- 后续可以更快生成设计任务或实现切片任务，但不会让 prompt-pattern 变成新的框架负担。
