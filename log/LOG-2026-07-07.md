# 2026-07-07 开发日志

本日志记录从空项目到 persistence 设计开始前的工作。这里按设计工作批次归档，不严格按 git 提交时间切分；execution-contract 的最终合成也归入本日志。persistence 设计不包含在内。

## 1. 空项目初始化

要解决的问题：

- 项目一开始没有固定结构，后续需求、设计、ADR、实现和验证很容易混在一起。
- AI agent 如果没有明确规则，可能会改无关文件、误提交临时文件，或者把设计任务直接写成实现代码。

实际做了什么：

- 初始化仓库，建立基础文件和文档目录。
- 建立 `docs/`、`docs/design/` 等文档承载位置。
- 保留 `LICENSE`、`.gitignore` 等基础项目文件。

结果：

- 项目从空目录变成一个可以持续沉淀需求、设计和后续实现的文档型仓库。

## 2. 协作规则与安全边界

要解决的问题：

- 这个项目会大量使用 AI agent 协作，需要先规定什么能做、什么不能做。
- 需要避免 provider adapter 访问数据库、日志泄露 API key、误把图片 BLOB 放进 SQLite、误做抓取/下载/分发功能等方向性错误。

实际做了什么：

- 编写 `AGENTS.md`，明确项目目标、权威文档、最小变更模式、设计优先规则、架构边界、数据模型不变量、质量和安全边界。
- 规定默认不 push、不 pull、不 rebase、不提交无关文件。
- 规定设计任务必须产出决策、理由、拒绝方案、风险、验证场景和开放问题。

结果：

- 后续每一轮设计都有统一约束，不会偏离“本地漫画翻译与基础嵌字工作流”这个产品边界。

## 3. 需求基线：SRS

要解决的问题：

- 在写架构前，需要先弄清楚产品到底给谁用、解决什么痛点、MVP 做到什么程度、明确不做什么。
- 如果没有需求边界，后面很容易变成专业汉化系统、漫画资源平台、复杂阅读器或通用工作流平台。

实际做了什么：

- 完成 `docs/SRS-v1.0.md`。
- 明确目标用户是普通漫画读者，不是专业汉化组。
- 明确第一阶段重点是日语到简体中文，目标是“基础可读”，不是发布级精修。
- 明确主流程：创建 Project、上传图片、检测文本块、OCR、翻译、质量检查、清字、嵌字、校对、局部返工、导出。
- 明确系统不提供漫画搜索、抓取、下载、分发、公开发布平台，也不绕过第三方 provider 策略。

结果：

- 形成正式需求基线，后续设计都围绕 MVP 一键处理、可恢复、可校对、可局部返工展开。

## 4. 概要设计：HLD

要解决的问题：

- 有了需求后，需要决定系统长什么样、模块怎么拆、谁负责什么。
- 需要提前避免“大对象包办一切”、UI 直接调用工具、provider 直接写数据库、artifact 生命周期散落各处等架构问题。

实际做了什么：

- 完成 `docs/HLD.md`，并补充 `docs/HLD-v0.2.md`。
- 确定应用形态：本地 Web UI + FastAPI 后端 + SQLite + 本地 filesystem workspace。
- 确定前端使用 React + Next.js，后端使用 Python + FastAPI，MVP 使用本地 TaskRunner。
- 拆出关键模块：WorkflowService、WorkflowLoopEngine、QualityCheckService、Provider Adapter、ArtifactService、Repository/DAO、ConfigService。
- 明确一键处理不是 API handler 同步跑完，而是创建 ProcessingTask 后由 TaskRunner 执行。
- 明确 Provider Adapter 只调用工具，不负责数据库、artifact 生命周期、重试、fallback、skip、block。

结果：

- 形成后续详细设计的架构基线：workflow 决策、质量检查、artifact 管理、provider 调用和持久化访问各有边界。

## 5. 项目推进规划：PROJECT-PLAN

要解决的问题：

- 项目不是一次性写完，需要知道先设计什么、先验证什么、什么时候再进入实现。
- OCR、翻译、清字、嵌字、provider refusal、中断恢复都有风险，不能等全部写完才发现跑不通。

实际做了什么：

- 编写 `docs/PROJECT-PLAN.md`。
- 选择“风险驱动的增量演化模型”：先做设计基线，再用 FakeProvider 验证 workflow、artifact、recovery 和 export gate。
- 把项目拆成阶段：需求与架构、核心详细设计、工具 spike、单 Page 后端垂直切片、单 Page Web 闭环、Batch/recovery/export、稳定化和后续演化。
- 明确当前阶段先完成 data-model、workflow-state、execution-contract，再进入 persistence 设计。

结果：

- 后续工作顺序变清楚：先把状态、数据、契约想清楚，再写 repository、DAO、migration 和 FakeProvider 单页切片。

## 6. Data Model 设计轮

要解决的问题：

- 这个系统必须可恢复、可解释、可局部重试。仅保存一个 Page 状态是不够的。
- 需要知道哪些数据放 `app.db`，哪些数据放每个 Project 自己的 `project.db`。
- 需要决定 OCR/translation 怎么版本化，用户编辑怎么追踪，当前生效结果怎么选择。
- 需要保证图片和大 payload 不进 SQLite，同时 artifact 又能被追踪、清理和恢复。

实际做了什么：

- 建立 `docs/design/data-model/GOAL.md`、`PLAN.md`、`HARNESS.md`。
- 生成 5 份 proposal，从领域模型、持久化、workflow state、artifact/quality、API/ORM readiness 角度分别设计。
- 做 cross review 和 harness validation，检查设计是否能覆盖成功路径、失败路径、用户编辑、provider refusal、清字失败、嵌字溢出、软删除、导出阻塞等场景。
- 合成最终设计：
  - `docs/design/data-model/final/data-model-dd-v0.1.md`
  - `docs/design/data-model/final/schema-outline.md`
  - `docs/design/data-model/final/state-data-impact.md`
  - `docs/design/data-model/final/erd.mmd`
  - `docs/design/data-model/final/open-questions.md`
- 记录 10 个 ADR，固定关键决策：
  - `app.db` / `project.db` 分离。
  - active pointer 作为当前结果来源。
  - artifact 只存元数据，文件在 filesystem。
  - OCRResult / TranslationResult 版本化。
  - WorkflowAttempt / WorkflowDecision 作为恢复和解释来源。
  - ProcessingProfileSnapshot 固化每次运行策略。
  - QualityIssue 控制 export gate。
  - Provider refusal 作为一等流程证据。
  - Project soft delete / trash 规则。
  - GlossaryVersion 支持翻译可复现。

结果：

- 数据模型不再只是“几张业务表”，而是能解释为什么失败、从哪里恢复、哪个结果当前有效、哪些 artifact 可以清理、为什么导出被阻止。
- 这个设计为后续 persistence/repository/migration 提供了输入，但没有写 SQL DDL 或 ORM。

## 7. Workflow State 设计轮

要解决的问题：

- 有了数据结构后，还需要知道 workflow 怎么推进。
- 系统遇到失败、超时、拒绝、低质量输出、用户编辑、崩溃恢复时，不能靠临时 if/else 猜。
- 需要定义 ProcessingTask、WorkflowAttempt、Page、TextBlock 各自的状态词汇，避免状态含义混乱。

实际做了什么：

- 建立 `docs/design/workflow-state/GOAL.md`、`PLAN.md`、`HARNESS.md`。
- 生成 5 份 proposal，分别覆盖状态词汇、决策重试、恢复幂等、stale propagation、实现就绪性。
- 做 cross review，确认没有阻塞性修订。
- 合成最终设计：
  - `docs/design/workflow-state/final/workflow-state-dd-v0.1.md`
  - `docs/design/workflow-state/final/state-vocabulary.md`
  - `docs/design/workflow-state/final/stage-transition-table.md`
  - `docs/design/workflow-state/final/decision-matrix.md`
  - `docs/design/workflow-state/final/recovery-rules.md`
  - `docs/design/workflow-state/final/stale-propagation-rules.md`
  - `docs/design/workflow-state/final/open-questions.md`
- 明确 MVP stage：检测、OCR、翻译、质量检查、清字、嵌字、导出就绪。
- 明确 retry、fallback、skip、warning、block、pause、cancel 等 decision 什么时候产生。
- 明确 OCR 编辑会让 translation/check/typeset stale，translation 编辑会让 typeset stale。
- 明确崩溃恢复优先看已提交结果、active pointer、attempt 和 artifact，而不是只看一个聚合状态。
- 记录 workflow-state ADR：状态词汇、retry/crash attempt、export warning readiness、恢复优先级。

结果：

- WorkflowLoopEngine 后续可以按表驱动思路实现，不需要把业务规则散在 provider、executor 或 API handler 里。
- 单页 FakeProvider 流程已经有明确的推进、失败、恢复和导出判断规则。

## 8. Execution Contract 设计轮

要解决的问题：

- data-model 和 workflow-state 说明了“要记录什么”和“流程怎么走”，但还缺少模块之间到底怎么交接。
- StageExecutor、Provider Adapter、ArtifactService、QualityCheckService、WorkflowLoopEngine 如果没有契约，后续实现很容易边界混乱。
- 需要让 FakeProvider 单 Page 后端垂直切片可实现，但不能提前接真实 OCR/翻译/清字/嵌字 provider。

实际做了什么：

- 建立 `docs/design/execution-contract/GOAL.md`、`PLAN.md`、`HARNESS.md`、`README.md`。
- 完成 preflight，确认 SRS、HLD、data-model final、workflow-state final 能作为输入。
- 生成 11 份 proposal：
  - Provider Adapter 边界和 DTO。
  - Provider 错误、拒绝、元数据和脱敏。
  - Provider capability 和 FakeProvider 行为。
  - ArtifactService 生命周期、taxonomy、恢复完整性。
  - QualityCheckService issue taxonomy、severity、root cause、用户可见行为。
  - StageExecutor 集成和 FakeProvider 垂直切片准备。
- 做 provider、artifact、quality 三个模块 debate。
- 做 cross-module contract review，检查模块之间有没有职责重叠或缺口。
- 做 no-blocking-revision 检查，然后合成最终设计：
  - `docs/design/execution-contract/final/execution-contract-dd-v0.1.md`
  - `docs/design/execution-contract/final/provider-adapter-contract.md`
  - `docs/design/execution-contract/final/artifact-service-contract.md`
  - `docs/design/execution-contract/final/quality-check-contract.md`
  - `docs/design/execution-contract/final/stage-executor-contract.md`
  - `docs/design/execution-contract/final/error-and-issue-taxonomy-minimal.md`
  - `docs/design/execution-contract/final/fakeprovider-readiness.md`
  - `docs/design/execution-contract/final/open-questions.md`
- 记录 execution-contract ADR：
  - provider result envelope 和 refusal 表示。
  - 临时 artifact promotion 到官方 artifact 的边界。
  - QualityCheck issue draft 与事务接纳边界。
  - StageExecutor 只产出证据，不做最终 workflow 决策。
  - FakeProvider deterministic modes。
  - redaction / sanitization 边界。

结果：

- 明确 Provider Adapter 返回什么、不返回什么、绝不能做什么。
- 明确 provider refusal 不是普通 crash，而是可记录、可决策、可 fallback/block 的流程证据。
- 明确 ArtifactService 是唯一官方 artifact 生命周期入口。
- 明确 QualityCheckService 只发现 issue 和归因，不推进 workflow。
- 明确 StageExecutor 执行一个 stage，收集 attempt/log/artifact/result/issue evidence，然后交给 WorkflowLoopEngine 决策。
- FakeProvider 单页后端切片需要模拟哪些 happy path 和 failure path 已经列清楚。

- 下一步 persistence 设计要解决 repository/DAO、migration、Unit of Work、事务边界、恢复查询和 FakeProvider 单页切片所需的最小持久化能力。
