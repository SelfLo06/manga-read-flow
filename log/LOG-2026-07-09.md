## 2. Slice 07：Idempotency and Recovery 真正落地

要解决的问题：

- 之前的 Slice 01-06 已经把 Project store、Repository/UoW、ArtifactService、FakeProvider、happy path、quality/readiness 铺好，但还没有真正把“重复运行”和“崩溃恢复”闭环打通。
- 需要证明系统不是靠 `Page.status` 或最新时间戳猜结果，而是靠 active pointer、attempt、decision、artifact 和 dependency hash 解释恢复。

实际做了什么：

- 在分支 `feat/slice-07-idempotency-recovery` 上完成 Slice 07 实现，提交为：
  - `374d57e feat(slice-07): add idempotency and recovery`
- 主要改动集中在：
  - `src/manga_read_flow/workflow/engine.py`
  - `src/manga_read_flow/workflow/reuse.py`
  - `src/manga_read_flow/workflow/readiness.py`
  - `src/manga_read_flow/persistence/acceptance_repository.py`
  - `src/manga_read_flow/persistence/content_state_repository.py`
  - `src/manga_read_flow/persistence/artifact_metadata_repository.py`
  - `src/manga_read_flow/providers/fake.py`
  - `src/manga_read_flow/artifacts/service.py`
  - `src/manga_read_flow/domain/artifacts.py`
  - `tests/integration/test_idempotency_and_recovery.py`

这一轮真正落地的行为是：

- unchanged rerun 可以复用 OCR / translation / cleaning / typesetting 结果；
- 复用行为是可审计的，不是悄悄跳过；
- OCR acceptance 后崩溃，恢复可以从 translation 继续；
- 已注册但未被 acceptance 选中的 artifact 不会因为“最新”就变成有效结果；
- missing active artifact 会被标记为 `missing`，然后交给 workflow 去决定 rebuild / warning / block；
- 恢复不再只看 `Page.status`。

结果：

- MVP-0 FakeProvider 单页后端的“重复运行 + 中断恢复”最后一块拼上了。
- Slice 07 的目标已经从计划层落到代码和集成测试层。
