# 1. 项目背景
看漫画时大部分资源都只有生肉，但是读者没有翻译能力，也没有修图能力，但是想看熟肉，对质量要求不苛刻，能做到80-90%的汉化组质量即可。在当下AI的发展和相关工具的发展下，这些需求可能能够实现。

# 2. 目标用户
没有汉化能力的纯读者，对于汉化重量要求不苛刻，只想快速看到熟肉漫画的普通读者。
# 3. 用户痛点
自己没有办法汉化，但是想看，不追求完美的汉化。
# 4. 产品目标
使用AI翻译和orc和修图工具，做到快速的翻译漫画，且达到汉化组80-90%的水平。
# 5. 核心使用场景
用户看到想看的漫画，上传下载的资源，汉化得到汉化完成的版本。
# 6. 功能性需求
以project为大单位，不同漫画分不同的project，一次能够汉化project内的一个批次的资源图片。
支持日语到中文，英语到中文的汉化。
同一project内共享术语表，不同project不影响。
用户可以导出汉化完的结果。
系统可以使用ocr识别每一页的文字，用户在ocr结束前看到ocr的效果，可选择修改或者重新识别某一页。
系统可以调用大模型api得到漫画内容的翻译，且用户可以选择对某一段重新做。
系统可以清除每一页的原文，用户在清除后查看清除的效果，可选择重新清除某一页的某些部分，或者退回重做某一页。
系统可以将汉化完的结果重新排版进入清除完的结果中，支持横排竖排，用户可以在导出前看到结果，且支持重排某一页或者某个部分。

# 7. 非功能性需求
汉化的速度不能太慢。
汉化的结果不能降低原图片质量。
汉化的结果在同一project内需要保持风格一致。
汉化的结果不能有太大的机翻味道和AI味道。
系统不能对设备性能要求太高。
系统不能太过臃肿，应该精简。
需要有一键得到结果的功能。
每个汉化批次需要能够从中断处恢复，支持手动暂停，也需要应对被动暂停。
机嵌的结果需要相对美观，不要超出气泡和硬横排，应该根据长度自适应。

# 8. MVP 范围
支持日语到中文，一键按照批次翻译得到结果，支持每一步可手动修改和返工，支持project和project内术语表。
# 9.  暂不支持范围
英文到中文功能，花体字和拟声词以及气泡外过于难识别的文字，以及给专业汉化组需要的更加细致的功能。
不在系统内提供完善的看图功能。
# 10.  风险与约束
资源可能nsfw，一些AI可能拒绝翻译，机翻机嵌可能不会受到大范围接受，仅能给个人解馋。
