# 130 OCR

OCR 对 Detection 区域生成版本化日文文本、置信与区域 provenance。用户编辑创建新的 OCRResult version；active pointer 选择当前结果，并使翻译及视觉下游按依赖标记 stale。

## Tool Role and Historical Pipeline

Primary Japanese recognition candidate 是 manga-ocr 0.1.14。维护者确认的历史实验链为：

```text
Paddle Detection
→ deterministic axis-aligned crop
→ manga-ocr Recognition
→ Grouping joins fragments
```

Paddle Recognition 只是备用候选，尚未冻结为正式 fallback。manga-ocr 在干净 crop 上的历史结论限定为 `CONDITIONAL_GO`；这不证明 Detection 碎片进入后的完整 OCR 质量，竖排、彩色、低对比、小字和碎片化仍是风险。

## Implementation Status

OCRResult version/pointer 和通用 Provider lifecycle 仅部分存在，`ProcessPageService` 当前仍使用 FakeProvider。正式 deterministic crop artifact/provenance、crop transform、real manga-ocr Adapter、真实 OCRCheck、正式产品入口和人工编辑闭环均未实现；现有 runner 位于 `tools/experiments/`，不能作为产品入口。

本机存在精确 Conda 环境 `manga-read-flow`：Python 3.12.13、PaddleOCR 3.7.0、paddlepaddle-gpu 3.3.1、manga-ocr 0.1.14、Torch 2.13.0+cu126。冻结 replay 已证明 Paddle 使用本地 `PP-OCRv6_medium_det` 缓存离线初始化；同一 replay 中 manga-ocr 默认初始化触发网络解析并被 network guard 阻止，实际 recognition 为 0。因此当前只记录环境与 Paddle capability，不把 manga-ocr local recognition 写成已证明事实，更不等同于正式 Adapter。

当前有真实/合成样本与人工 GT，可用于工具验证；尚未证明自动 OCR 在产品支持范围内达到 M2 系统性质量。M1 仍要求真实本地 OCR 可运行、问题可发现、可人工修正，且普通正文不得静默遗漏。

拒绝覆盖原 OCR、把人工 GT 当运行时输入、伪造 confidence/character geometry/orientation，或将低质量结果静默接受。验证覆盖空白/多行、低置信或不可判定质量、拒绝、人工编辑、版本复用、stale propagation 和真实入口；M2 系统性阈值仍待冻结。
