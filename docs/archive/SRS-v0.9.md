## 1. 引言

### 1.1 编写目的

本文档用于定义“漫画翻译与基础嵌字自动化工作流应用”的软件需求规格，明确系统的目标用户、使用场景、功能性需求、非功能性需求、数据需求、处理流程、MVP 范围和暂不支持范围。

本文档的目标不是确定具体技术实现方案，而是作为后续工具可行性调研、概要设计、详细设计和 MVP 开发的需求基线。

后续将基于 OCR、大模型翻译、文字清除、图像修复、嵌字排版等外部工具的可行性调研结果，对本文档进行修订，形成 SRS v1.0。

### 1.2 项目背景

大量漫画资源仅存在原语言版本。普通读者在不具备外语阅读能力、修图能力和汉化经验的情况下，难以直接阅读这些漫画内容。

传统汉化流程通常包括翻译、校对、修图、清字、嵌字、排版等多个步骤，对普通读者门槛较高。普通读者通常并不追求专业汉化组级别的发布质量，而是更关心能否快速理解漫画剧情、人物对话和主要信息。

随着 OCR、图像修复、大语言模型翻译和自动排版技术的发展，面向普通读者的自动化漫画翻译与基础嵌字工作流具备一定可行性。

本项目希望将这些能力整合为一个相对简单、可恢复、可局部返工的个人阅读辅助工具。

### 1.3 产品目标

本系统旨在构建一个面向普通读者的漫画翻译与基础嵌字自动化工作流应用。

系统以 Project 为单位组织不同漫画作品。用户可以在 Project 内上传漫画图片批次，系统自动完成普通文本区域检测、OCR 识别、大模型翻译、基础清字、基础嵌字和结果导出。

系统第一优先级支持日语到中文。英语到中文作为后续扩展方向，不纳入 MVP 阶段。

系统不以专业汉化发布为目标，而是优先保证普通读者能够较顺畅地理解漫画正文内容。

### 1.4 适用范围

本文档适用于以下后续工作：

* 工具可行性调研；
* 系统概要设计；
* 系统详细设计；
* MVP 功能拆分；
* 数据模型设计；
* 任务状态机设计；
* 前后端功能开发；
* 测试用例设计；
* 后续 SRS v1.0 修订。

### 1.5 术语定义

| 术语        | 定义                                              |
| --------- | ----------------------------------------------- |
| 生肉        | 未经过中文翻译和汉化处理的原语言漫画资源                            |
| 熟肉        | 已经过中文翻译和嵌字处理的漫画资源                               |
| Project   | 系统中用于表示一部漫画或一个独立漫画翻译任务集合的业务单位                   |
| Batch     | Project 内一次上传和处理的图片集合，可对应一话漫画、一个章节或一组图片         |

| TextBlock | 页面中的一个文本区域，例如对话框文字、旁白框文字                        |

| MVP       | Minimum Viable Product，最小可用产品                   |

## 2. 总体描述

### 2.1 产品定位

本系统是一个面向普通漫画读者的个人阅读辅助工具。

它不是专业汉化组生产工具，不追求发布级精修效果，也不负责漫画资源搜索、抓取、下载、分发或发布。

系统重点解决的问题是：

用户拥有一批原语言漫画图片，但无法直接阅读。系统帮助用户将这些图片自动转换为基础中文可读版本，并允许用户进行必要的简单修正。

### 2.2 目标用户

#### 2.2.1 核心用户：普通读者

普通读者是本系统的第一目标用户。

核心用户特征：

* 不具备稳定的日语或英语漫画阅读能力；
* 不具备专业修图和嵌字能力；
* 对发布级汉化质量没有强需求；
* 更关心能否快速看懂剧情和对话；
* 能接受部分复杂文字区域不被处理；
* 希望操作流程简单；
* 希望一键得到可读结果；
* 希望结果不满意时可以做简单修改。

#### 2.2.2 次要用户：个人汉化爱好者

个人汉化爱好者是本系统的次要用户。

这类用户可能希望基于自动处理结果进行简单人工校对，包括修改 OCR、修改译文、调整文本框、局部重跑等。

系统暂不面向专业汉化组，不提供复杂协作、发布审核、多角色分工、精修排版、复杂拟声词重绘等功能。

### 2.3 用户痛点

普通读者的主要痛点包括：

* 想看原语言漫画，但没有足够外语能力；
* 逐张截图翻译效率低，阅读体验割裂；
* OCR、翻译、清字、嵌字工具分散，普通用户难以组合使用；
* 人名、术语、技能名在不同页面之间容易翻译不一致；
* 整章漫画处理时间可能较长，中断后如果需要从头处理，体验很差；
* 专业汉化流程质量高，但门槛过高；
* 普通读者主要需要可读版本，不一定需要处理复杂拟声词、花体字和艺术字。

### 2.4 产品主流程

系统主流程如下：

```text
创建 Project
→ 在 Project 内创建 Batch
→ 上传漫画图片
→ 选择源语言
→ 启动自动处理
→ 文本区域检测
→ OCR 识别
→ 调用术语表和大模型进行翻译
→ 清除原文
→ 自动嵌入中文译文
→ 保存中间结果
→ 用户查看与校对
→ 局部返工
→ 导出结果
```

### 2.5 设计约束

系统存在以下约束：

* MVP 阶段优先支持日语到中文；
* MVP 阶段重点处理普通对话框和旁白框文字；
* MVP 阶段不处理复杂拟声词、花体字、艺术字和复杂背景文字；
* 系统依赖外部 OCR、翻译、清字、嵌字或图像处理能力；
* 外部工具可能存在识别错误、翻译错误、API 成本、速度限制、内容限制和服务不可用等问题；
* 系统应通过状态持久化和局部重试降低外部工具失败造成的影响；
* 系统不提供漫画资源获取能力；
* 系统不提供公开发布或分发能力。

### 2.6 假设条件

本文档基于以下假设：

* 用户已合法持有或自行准备待处理漫画图片；
* 用户上传的图片格式为常见图片格式；
* 用户主要处理日语漫画；
* 用户对结果的首要要求是“可理解”，其次才是“美观”；
* 用户能接受部分复杂文字区域被跳过；
* 外部 OCR、翻译、图像处理和排版工具具备基本可用性；
* 具体工具可行性需要在 SRS v0.9 后进行验证。

## 3. 系统范围

### 3.1 系统包含的能力

MVP 阶段系统包含以下能力：

* Project 管理；
* Batch 管理；
* 图片上传与页面管理；
* 日语到中文处理流程；
* 普通文本区域检测；
* OCR 识别；
* 大模型翻译；
* Project 内术语表；
* 基础清字；
* 基础嵌字；
* 处理状态记录；
* 中断恢复；
* 局部重试；
* 简单校对；
* 图片或 ZIP 导出。

### 3.2 系统不包含的能力

MVP 阶段系统不包含以下能力：

* 英语到中文完整处理流程；
* 其他语言到中文；
* 漫画资源搜索；
* 漫画资源抓取；
* 漫画资源下载；
* 漫画资源分发；
* 发布平台；
* 多人协作；
* 专业汉化组流程；
* 复杂拟声词重绘；
* 花体字和艺术字重绘；
* 复杂背景文字精修；
* 完整漫画阅读器；
* 大规模云端任务调度；
* 精细版本管理。

## 4. 功能性需求

### 4.1 Project 管理

#### FR-PJ-001 创建 Project

系统应支持用户创建 Project。

每个 Project 表示一部漫画或一个独立漫画翻译任务集合。

#### FR-PJ-002 查看 Project

系统应支持用户查看已有 Project 列表。

Project 列表应至少展示：

* Project 名称；
* 最近处理时间；
* 批次数量；
* 当前处理状态摘要。

#### FR-PJ-003 编辑 Project

系统应支持用户重命名 Project。

#### FR-PJ-004 删除 Project

系统应支持用户删除 Project。

删除 Project 时，系统应提示用户该操作会删除该 Project 下的图片、批次、术语表、中间结果和导出记录。

#### FR-PJ-005 Project 数据隔离

不同 Project 之间的数据应相互隔离。

隔离的数据包括：

* 图片；
* Batch；
* Page；
* TextBlock；
* OCR 结果；
* 翻译结果；
* 清字结果；
* 嵌字结果；
* 术语表；
* 处理状态；
* 导出结果。

#### FR-PJ-006 Project 内共享上下文

同一 Project 内的批次应共享该 Project 的术语表。

后续可扩展为共享角色名、历史译法和翻译风格设置。

---

### 4.2 Batch 管理

#### FR-BT-001 创建 Batch

用户应能在 Project 内创建 Batch。

一个 Batch 表示一次上传和处理的漫画图片集合。

#### FR-BT-002 上传图片

用户应能向 Batch 上传一张或多张漫画图片。

系统应支持以下图片格式：

* PNG；
* JPG；
* JPEG；
* WebP。

#### FR-BT-003 保留图片顺序

系统应保留用户上传图片的页面顺序。

如果系统无法判断顺序，应允许用户手动调整页面顺序。

#### FR-BT-004 保留原始图片

系统应保存用户上传的原始图片。

后续 OCR、清字、嵌字和导出结果不得覆盖原始图片。

#### FR-BT-005 删除 Batch

用户应能删除 Batch。

删除 Batch 时，系统应提示用户该操作会删除该 Batch 下的页面、中间结果和导出结果。

#### FR-BT-006 查看 Batch 状态

用户应能查看 Batch 的整体处理状态。

Batch 状态应能反映当前批次中页面处理进度、失败数量和是否已完成。

---

### 4.3 Page 管理

#### FR-PG-001 Page 创建

系统应为 Batch 中的每张图片创建 Page 记录。

#### FR-PG-002 Page 状态查看

用户应能查看每个 Page 的处理状态。

状态应至少包括：

* 已上传；
* 文本检测完成；
* OCR 完成；
* 翻译完成；
* 清字完成；
* 嵌字完成；
* 等待校对；
* 已导出；
* 失败；
* 已暂停。

#### FR-PG-003 Page 预览

用户应能查看 Page 的以下内容：

* 原图；
* 检测到的文本区域；
* OCR 结果；
* 翻译结果；
* 清字结果；
* 嵌字结果。

MVP 阶段不要求提供完整漫画阅读器功能，只要求满足处理预览和校对需要。

#### FR-PG-004 Page 删除

用户应能删除 Batch 中的单张 Page。

#### FR-PG-005 Page 重新处理

用户应能对单张 Page 重新执行部分或全部处理流程。

---

### 4.4 语言选择

#### FR-LANG-001 源语言选择

用户在创建 Batch 或启动处理前，应能选择源语言。

MVP 阶段源语言应至少支持：

* 日语。

#### FR-LANG-002 目标语言

MVP 阶段目标语言固定为中文。

#### FR-LANG-003 英语后续扩展

英语到中文作为后续扩展能力，不纳入 MVP 阶段的必须实现范围。

---

### 4.5 文本区域检测

#### FR-DET-001 自动检测文本区域

系统应自动检测 Page 中的普通文本区域。

MVP 阶段重点检测：

* 普通对话框；
* 旁白框；
* 清晰印刷体文字。

#### FR-DET-002 TextBlock 创建

系统应为检测到的每个文本区域创建 TextBlock 记录。

TextBlock 应至少包含：

* 所属 Project；
* 所属 Batch；
* 所属 Page；
* 区域坐标；
* 区域大小；
* 文本方向；
* 检测置信度；
* 处理状态。

#### FR-DET-003 文本方向记录

系统应记录文本方向。

文本方向至少包括：

* 横排；
* 竖排；
* 未确定。

#### FR-DET-004 阅读顺序

系统应尽量判断同一 Page 内 TextBlock 的阅读顺序。

如果无法可靠判断，系统应允许用户手动调整顺序，或在后续版本中提供调整能力。

#### FR-DET-005 复杂区域跳过

对于复杂拟声词、花体字、艺术字、严重倾斜文字、复杂背景文字和难以识别文字，系统允许跳过处理。

被跳过区域应尽量保留原图内容，不应导致整页处理失败。

#### FR-DET-006 检测结果修改

用户应能查看检测结果。

P1 阶段用户应能手动调整 TextBlock 区域。

---

### 4.6 OCR 识别

#### FR-OCR-001 OCR 执行

系统应对检测到的 TextBlock 执行 OCR 识别。

#### FR-OCR-002 OCR 结果保存

系统应保存每个 TextBlock 的 OCR 结果。

OCR 结果应至少包含：

* 识别文本；
* 识别置信度；
* OCR 工具或模型来源；
* OCR 时间；
* 是否被用户修改。

#### FR-OCR-003 OCR 结果查看

用户应能查看每个 TextBlock 的 OCR 识别结果。

#### FR-OCR-004 OCR 结果修改

用户应能手动修改 OCR 识别结果。

修改后的 OCR 文本应作为后续翻译依据。

#### FR-OCR-005 单块 OCR 重试

用户应能对单个 TextBlock 重新执行 OCR。

#### FR-OCR-006 单页 OCR 重试

用户应能对单张 Page 重新执行 OCR。

#### FR-OCR-007 OCR 失败处理

单个 TextBlock OCR 失败不应导致整张 Page 失败。

单张 Page OCR 部分失败时，系统应保留已成功识别的 TextBlock 结果。

---

### 4.7 大模型翻译

#### FR-TR-001 翻译执行

系统应调用大模型翻译模块，将 OCR 结果翻译为中文。

#### FR-TR-002 使用 Project 术语表

翻译时系统应读取当前 Project 的术语表。

术语表中的固定译法应作为翻译约束传入翻译模块。

#### FR-TR-003 翻译上下文

系统应尽量向翻译模块提供必要上下文。

上下文可包括：

* 同一 Page 内的其他文本；
* TextBlock 阅读顺序；
* 当前 Project 术语表；
* 已确认的人名和术语；
* 用户修改过的译文。

#### FR-TR-004 翻译结果保存

系统应保存每个 TextBlock 的翻译结果。

翻译结果应至少包含：

* 源文本；
* 译文；
* 使用的术语表版本；
* 翻译工具或模型来源；
* 翻译时间；
* 是否被用户修改。

#### FR-TR-005 翻译结果查看

用户应能查看每个 TextBlock 的源文本和译文。

#### FR-TR-006 翻译结果修改

用户应能手动修改译文。

#### FR-TR-007 单块重新翻译

用户应能对单个 TextBlock 重新翻译。

#### FR-TR-008 单页重新翻译

用户应能对单张 Page 重新翻译。

#### FR-TR-009 批次重新翻译

用户应能对整个 Batch 重新翻译。

批次重新翻译应明确提示可能产生额外耗时和模型调用成本。

#### FR-TR-010 译文质量要求

翻译结果应优先保证普通中文读者能够理解剧情和对话。

翻译应尽量避免：

* 明显直译；
* 语义不通顺；
* 人名漂移；
* 术语漂移；
* 称呼前后不一致；
* 过度模板化表达；
* 明显 AI 腔。

#### FR-TR-011 翻译失败处理

翻译失败时，系统应记录失败原因。

单个 TextBlock 翻译失败不应导致整张 Page 其他 TextBlock 结果丢失。

---

### 4.8 Project 术语表

#### FR-GL-001 独立术语表

每个 Project 应拥有独立术语表。

不同 Project 的术语表不得自动互相影响。

#### FR-GL-002 术语字段

术语表条目应至少包含：

* 原文；
* 译文；
* 类型；
* 备注；
* 创建时间；
* 更新时间。

#### FR-GL-003 术语类型

术语类型应至少支持：

* 人名；
* 地名；
* 组织名；
* 技能名；
* 专有名词；
* 固定表达；
* 其他。

#### FR-GL-004 新增术语

用户应能在 Project 内新增术语。

#### FR-GL-005 编辑术语

用户应能编辑已有术语。

#### FR-GL-006 删除术语

用户应能删除术语。

#### FR-GL-007 从译文添加术语

用户修改译文后，系统应支持用户将某个词语或短语快速加入当前 Project 的术语表。

#### FR-GL-008 术语表版本

系统应记录翻译时使用的术语表状态。

后续可扩展为术语表版本管理。

---

### 4.9 清字与图像修复

#### FR-CL-001 清字执行

系统应根据 TextBlock 区域位置清除原文。

#### FR-CL-002 处理范围

MVP 阶段清字主要处理：

* 普通对话框文字；
* 旁白框文字；
* 背景相对简单的文字区域。

#### FR-CL-003 保留原图

清字过程不得覆盖原始图片。

#### FR-CL-004 清字结果保存

系统应保存清字后的中间图片结果。

#### FR-CL-005 清字结果查看

用户应能查看清字结果。

#### FR-CL-006 单块重新清字

用户应能对单个 TextBlock 区域重新清字。

#### FR-CL-007 单页重新清字

用户应能对单张 Page 重新清字。

#### FR-CL-008 复杂区域跳过

对于复杂背景文字、拟声词、花体字、艺术字等区域，系统可以默认跳过清字。

跳过区域不应导致整页处理失败。

#### FR-CL-009 清字失败处理

清字失败时，系统应保留原图和已有中间结果，并允许用户重新执行清字。

---

### 4.10 嵌字与排版

#### FR-TY-001 嵌字执行

系统应将中文译文嵌入清字后的图片中。

#### FR-TY-002 基础横排嵌字

MVP 阶段系统应支持基础中文横排嵌字。

#### FR-TY-003 竖排文本处理

系统应记录原文本方向。

对于原竖排文本，MVP 阶段可先采用基础中文横排嵌字或后续调研后确定是否支持竖排中文嵌字。

#### FR-TY-004 自动换行

系统应根据 TextBlock 区域大小对译文自动换行。

#### FR-TY-005 字号自适应

系统应根据译文长度和文本区域大小自动调整字号。

#### FR-TY-006 边界约束

嵌字结果应尽量避免超出气泡、旁白框或目标文本区域边界。

#### FR-TY-007 避免硬横排

系统应尽量避免出现明显影响阅读的硬横排效果。

#### FR-TY-008 排版结果保存

系统应保存嵌字后的 Page 结果。

#### FR-TY-009 排版结果查看

用户应能查看嵌字结果。

#### FR-TY-010 单块重新排版

用户应能对单个 TextBlock 重新排版。

#### FR-TY-011 单页重新排版

用户应能对单张 Page 重新排版。

#### FR-TY-012 手动调整

P1 阶段用户应能调整：

* 文本框位置；
* 字号；
* 换行；
* 排版方向；
* 字体样式。

---

### 4.11 校对与返工

#### FR-RV-001 结果校对界面

系统应提供校对界面，使用户能够查看：

* 原图；
* TextBlock 区域；
* OCR 原文；
* 中文译文；
* 清字结果；
* 嵌字结果。

#### FR-RV-002 修改 OCR

用户应能修改 OCR 文本。

#### FR-RV-003 修改译文

用户应能修改翻译结果。

#### FR-RV-004 局部返工

用户应能对局部内容重新处理。

局部返工粒度至少包括：

* 单个 TextBlock；
* 单张 Page。

#### FR-RV-005 避免全量重跑

除非用户主动要求，系统不应因为局部修改而重新处理整个 Batch。

#### FR-RV-006 修改后联动

当用户修改 OCR 文本后，系统应允许用户重新翻译相关 TextBlock。

当用户修改译文后，系统应允许用户重新排版相关 TextBlock。

---

### 4.12 任务状态与中断恢复

#### FR-TS-001 状态持久化

系统应持久化保存 Batch、Page 和 TextBlock 的处理状态。

#### FR-TS-002 中间结果持久化

系统应保存关键中间结果，包括：

* 检测结果；
* OCR 结果；
* 翻译结果；
* 清字结果；
* 嵌字结果。

#### FR-TS-003 手动暂停

用户应能手动暂停正在处理的 Batch。

#### FR-TS-004 被动中断恢复

系统应支持异常关闭、网络中断、外部 API 失败等情况下的恢复。

#### FR-TS-005 继续处理

用户重新打开 Project 或 Batch 后，系统应显示上次处理进度，并允许用户从未完成步骤继续处理。

#### FR-TS-006 单页失败隔离

单张 Page 失败不应导致整个 Batch 失败。

#### FR-TS-007 单块失败隔离

单个 TextBlock 失败不应导致整张 Page 已完成结果丢失。

#### FR-TS-008 避免重复调用

已完成且未被用户要求重做的步骤不应重复调用 OCR、翻译、清字或嵌字工具。

#### FR-TS-009 失败重试

系统应允许用户对失败的 Page 或 TextBlock 进行重试。

### 4.13 导出

#### FR-EX-001 单页导出

用户应能导出单张 Page 的处理结果。

#### FR-EX-002 批次导出

用户应能导出整个 Batch 的处理结果。

#### FR-EX-003 ZIP 导出

系统应支持将 Batch 结果导出为 ZIP 压缩包。

#### FR-EX-004 页面顺序

导出结果应保持 Page 顺序。

#### FR-EX-005 图片质量

导出图片应尽量保持原始分辨率和清晰度。

#### FR-EX-006 可编辑数据保留

导出结果后，系统仍应保留 Project 内的可编辑数据，方便用户后续继续修改。

## 5. 非功能性需求

### 5.1 可用性

#### NFR-USE-001 一键处理

系统应提供一键处理入口，使普通读者不需要理解 OCR、模型参数、图像修复参数和排版参数。

#### NFR-USE-002 进度展示

系统应展示处理进度。

进度信息应至少包括：

* 当前处理阶段；
* 当前处理页面；
* 已完成页面数量；
* 失败页面数量；
* 是否暂停；
* 是否完成。

#### NFR-USE-003 错误提示

系统应在失败时给出用户可理解的错误提示。

错误提示不应只显示底层异常信息。

#### NFR-USE-004 简单优先

MVP 界面应优先服务普通读者，不应将复杂参数暴露在主流程中。

高级设置可以后续扩展，但不应阻碍一键处理。

---

### 5.2 可靠性

#### NFR-REL-001 中断恢复

系统应在任务中断后恢复到最近可继续处理的状态。

#### NFR-REL-002 局部失败隔离

系统应避免单个 Page 或 TextBlock 失败导致整个 Batch 报废。

#### NFR-REL-003 数据不丢失

系统应尽量避免因应用关闭、网络失败或外部工具失败导致已完成结果丢失。

#### NFR-REL-004 原图安全

系统不得覆盖或破坏用户上传的原始图片。

---

### 5.3 结果质量

#### NFR-QLT-001 可读性优先

系统输出结果应优先保证普通读者能够理解剧情和对话。

#### NFR-QLT-002 术语一致性

系统应尽量保持同一 Project 内人名、地名、技能名、专有名词和称呼一致。

#### NFR-QLT-003 译文自然度

系统应尽量避免明显生硬直译、语义不通顺和 AI 模板化表达。

#### NFR-QLT-004 图像质量

系统应尽量避免降低原图分辨率和清晰度。

#### NFR-QLT-005 嵌字可读性

嵌字结果应尽量避免：

* 文字超出气泡；
* 字号过小；
* 换行严重不自然；
* 中文排版影响理解；
* 文字覆盖重要画面信息。

#### NFR-QLT-006 非发布级质量声明

系统不承诺生成发布级汉化成品。

MVP 阶段目标是生成个人阅读可用的基础汉化结果。

---

### 5.4 性能

#### NFR-PERF-001 异步处理

系统应支持 Batch 异步处理，避免用户界面长时间阻塞。

#### NFR-PERF-002 进度反馈

当处理耗时较长时，系统应持续提供进度反馈。

#### NFR-PERF-003 避免重复处理

系统应复用已保存的中间结果，避免不必要的重复 OCR、翻译、清字和嵌字。

#### NFR-PERF-004 局部处理

系统应支持局部重新处理，以降低用户等待时间和外部工具调用成本。

---

### 5.5 成本控制

#### NFR-COST-001 缓存翻译结果

系统应缓存翻译结果。

除非用户主动要求重新翻译，否则不应重复调用大模型处理相同文本。

#### NFR-COST-002 缓存 OCR 结果

系统应缓存 OCR 结果。

除非用户主动要求重新识别，否则不应重复 OCR 已完成的 TextBlock。

#### NFR-COST-003 局部重试

系统应支持单页或单块重试，避免因局部错误造成整批次重跑。

#### NFR-COST-004 成本提示

后续版本可在用户执行批量重翻译、批量重清字等高成本操作前给出提示。

---

### 5.6 可维护性

#### NFR-MNT-001 模块解耦

系统应将以下模块解耦：

* Project 管理；
* Batch 管理；
* 图片管理；
* 文本检测；
* OCR；
* 翻译；
* 术语表；
* 清字；
* 嵌字；
* 校对；
* 导出；
* 任务调度。

#### NFR-MNT-002 外部工具可替换

OCR、翻译、清字和嵌字模块应通过接口封装，尽量避免与具体工具强绑定。

#### NFR-MNT-003 错误记录

系统应记录任务处理错误，方便定位失败原因。

#### NFR-MNT-004 状态可追踪

系统应能追踪 Batch、Page 和 TextBlock 的处理状态。

---

### 5.7 可扩展性

#### NFR-EXT-001 多语言扩展

系统应保留后续支持英语到中文和其他语言到中文的扩展空间。

#### NFR-EXT-002 多模型扩展

系统应保留后续接入不同 OCR、LLM、图像修复和排版工具的扩展空间。

#### NFR-EXT-003 高级编辑扩展

系统应保留后续增强文本框编辑、字体设置、竖排嵌字和更复杂清字能力的扩展空间。

---

### 5.8 内容与合规约束

#### NFR-COMP-001 用户主动上传

系统只处理用户主动上传的图片。

系统不提供漫画资源搜索、抓取、下载或分发功能。

#### NFR-COMP-002 第三方 API 限制

如果系统使用第三方模型 API，用户上传内容可能受到第三方服务条款和内容策略限制。

#### NFR-COMP-003 内容处理失败

对于可能被 OCR、翻译或图像处理工具拒绝处理的内容，系统应提示用户处理失败，而不应尝试绕过外部服务限制。

#### NFR-COMP-004 NSFW 风险

用户上传内容可能包含 NSFW 或其他敏感内容，部分外部工具可能拒绝处理。

系统应将其视为外部依赖风险，而不是承诺必然可处理。

#### NFR-COMP-005 发布风险

MVP 阶段不提供公开发布、分享或分发功能。

如果后续增加发布能力，需要重新评估版权和内容合规风险。

## 7. 状态设计

### 7.1 Batch 状态

Batch 状态应至少包括：

```text
created          已创建
uploaded         已上传
processing       处理中
paused           已暂停
partially_failed 部分失败
failed           全部失败
reviewing        等待查看或校对
completed        处理完成
exported         已导出
```

### 7.2 Page 状态

Page 状态应至少包括：

```text
uploaded       已上传
detected       文本检测完成
ocr_done       OCR 完成
translated     翻译完成
cleaned        清字完成
typeset_done   嵌字完成
reviewing      等待查看或校对
exported       已导出
failed         处理失败
skipped        已跳过
```

### 7.3 TextBlock 状态

TextBlock 状态应至少包括：

```text
detected       已检测
ocr_done       OCR 完成
translated     翻译完成
cleaned        清字完成
typeset_done   嵌字完成
edited         用户已修改
failed         处理失败
skipped        已跳过
```

### 7.4 状态恢复规则

系统恢复任务时，应根据状态判断下一步处理阶段。

规则示例：

* Page 状态为 `uploaded` 时，应从文本检测开始；
* Page 状态为 `detected` 时，应从 OCR 开始；
* Page 状态为 `ocr_done` 时，应从翻译开始；
* Page 状态为 `translated` 时，应从清字开始；
* Page 状态为 `cleaned` 时，应从嵌字开始；
* Page 状态为 `typeset_done` 时，应进入查看或校对阶段；
* Page 状态为 `failed` 时，应允许用户重试失败步骤；
* TextBlock 状态为 `skipped` 时，不应阻塞 Page 其他 TextBlock 的处理。

## 8. 外部依赖与技术约束

### 8.1 OCR 依赖

系统依赖 OCR 能力完成漫画文本识别。

OCR 模块应尽量满足以下能力：

* 支持日语识别；
* 支持漫画图片文本识别；
* 能返回文本内容；
* 能返回文本区域坐标；
* 能返回置信度；
* 能处理横排和部分竖排文本。

具体 OCR 工具将在工具可行性调研阶段确定。

### 8.2 大模型翻译依赖

系统依赖大模型能力完成翻译。

翻译模块应尽量满足以下能力：

* 支持日语到中文；
* 支持传入上下文；
* 支持传入 Project 术语表；
* 能保持人名和术语一致；
* 能输出自然中文译文；
* 支持局部重试。

具体模型或 API 将在工具可行性调研阶段确定。

### 8.3 清字与图像修复依赖

系统依赖文字清除或图像修复能力完成清字。

清字模块应尽量满足以下能力：

* 根据 TextBlock 区域生成 mask；
* 清除普通对话框或旁白框中的原文；
* 尽量保持背景自然；
* 支持单块或单页重试；
* 输出可继续嵌字的中间图片。

具体清字工具将在工具可行性调研阶段确定。

### 8.4 嵌字与排版依赖

系统依赖嵌字排版能力完成中文文本绘制。

嵌字模块应尽量满足以下能力：

* 支持中文字体绘制；
* 支持自动换行；
* 支持字号自适应；
* 支持基础横排；
* 尽量避免文字溢出；
* 支持单块或单页重排。

具体嵌字工具或自研方案将在工具可行性调研和设计阶段确定。

### 8.5 外部工具失败约束

外部工具可能出现以下失败：

* OCR 识别失败；
* OCR 识别错误；
* 翻译 API 拒绝处理；
* 翻译结果质量差；
* 清字效果差；
* 嵌字溢出；
* 网络失败；
* 调用超时；
* 费用或额度不足。

系统应通过状态保存、失败重试、局部返工和用户手动修改降低这些问题造成的影响。

## 9. MVP 范围

### 9.1 MVP 必须包含

MVP 阶段应包含：

* Project 创建、查看、重命名、删除；
* Project 内独立术语表；
* Batch 创建；
* 多图片上传；
* Page 顺序管理；
* 日语到中文；
* 普通文本区域检测；
* OCR 识别；
* 大模型翻译；
* 基础清字；
* 基础嵌字；
* 处理结果预览；
* 修改 OCR；
* 修改译文；
* 单页重试；
* 单个 TextBlock 重试；
* 中断恢复；
* 图片导出；
* ZIP 导出。

* 英语到中文完整流程；

* 复杂拟声词处理；

### 9.3 MVP 成功标准

MVP 可认为成功，如果满足以下条件：

* 用户能够创建 Project；
* 用户能够上传一批日语漫画图片；
* 系统能够生成基础中文可读结果；
* 用户能够查看 OCR、译文和嵌字结果；
* 用户能够修改明显错误；
* 用户能够从中断处继续处理；
* 用户能够导出处理结果；
* 复杂拟声词、花体字和复杂背景文字不会阻塞主流程。

## 11. 风险与约束

### 11.1 技术风险

* OCR 对竖排日文、低清图片、花体字和复杂背景文字识别效果可能不稳定；
* 大模型可能出现错译、漏译、称呼不一致和术语漂移；
* 大模型可能因为内容策略拒绝处理部分文本；
* 清字可能破坏背景；
* 嵌字可能出现字号过小、换行不自然或文字溢出；
* 整批次处理耗时可能较长；
* 外部工具调用可能失败或超时；
* 外部工具成本可能影响使用体验。

### 11.2 产品风险

* 用户对“可读”的接受标准存在差异；
* 如果自动结果质量过低，用户可能认为校对成本过高；
* 如果交互流程过复杂，普通读者可能失去使用意愿；
* 如果功能范围过度扩张，个人项目开发压力会显著增加；
* 如果过早追求专业汉化效果，MVP 可能难以完成。

### 11.3 内容风险

* 用户上传内容可能包含 NSFW 或其他敏感内容，但系统不应该拒绝；
* 外部模型或 API 可能拒绝处理部分内容；
* 系统不提供资源分发能力；

### 11.4 资源约束

* 本项目为个人兴趣项目，开发资源有限；
* MVP 阶段应优先服务普通读者核心阅读需求；
* 不应在早期投入过多专业汉化组功能；
* 需要在质量、速度、成本和实现难度之间进行取舍。

## 12. 验收标准

### 12.1 Project 验收

* 用户可以创建 Project；
* 用户可以查看 Project；
* 用户可以删除 Project；
* 不同 Project 术语表互不影响。

### 12.2 Batch 验收

* 用户可以在 Project 内上传多张图片；
* 系统可以保持图片顺序；
* 系统可以显示 Batch 处理状态；
* 用户可以暂停和继续 Batch。

### 12.3 OCR 验收

* 系统可以对普通对话框或旁白框中的日文文本进行 OCR；
* 用户可以查看 OCR 结果；
* 用户可以修改 OCR 结果；
* OCR 失败不导致整个 Batch 报废。

### 12.4 翻译验收

* 系统可以将 OCR 结果翻译为中文；
* 翻译时可以参考 Project 术语表；
* 用户可以修改译文；
* 用户可以对单个 TextBlock 或单张 Page 重新翻译。

### 12.5 清字验收

* 系统可以对普通文字区域执行基础清字；
* 清字结果不覆盖原图；
* 用户可以查看清字结果；
* 清字失败时可以重试。

### 12.6 嵌字验收

* 系统可以将中文译文嵌入图片；
* 嵌字结果尽量不超出文本区域；
* 用户可以查看嵌字结果；
* 用户可以对局部内容重新排版。

### 12.7 中断恢复验收

* 处理中断后，系统可以保留已完成结果；
* 用户重新打开 Batch 后，可以从未完成步骤继续；
* 已完成步骤不会被无故重复处理。

### 12.8 导出验收

* 用户可以导出单页结果；
* 用户可以导出整个 Batch；
* 导出结果保持页面顺序；
* 导出图片尽量保持原图分辨率。
